1. The mental model

Every web app like this is three boxes talking to each other:

flowchart LR A["Frontend
(React in the browser)
what the user sees & clicks"] -->|"HTTP request
(fetch)"| B["Backend / API
(Express on a server)
rules, auth, logic"] B -->|"SQL query
(via Prisma)"| C["Database
(Postgres)
stores the data forever"] C -->|"rows"| B B -->|"JSON response"| A
  • Frontend — runs in the user's browser. Draws the UI, handles clicks, but owns no data. When it needs data it asks the backend.
  • Backend (API) — runs on a server you control. It's the gatekeeper: checks who you are (auth), what you're allowed to do (RBAC), validates input, then reads/writes the database.
  • Database — the permanent memory. Everything else can restart and lose nothing because the truth lives here.
Why separate them? The browser can't be trusted (users can edit it, forge requests). So the browser never touches the database directly — it must go through the backend, which enforces all the rules. This separation is the single most important idea in web security.
Today vs. where we're going. Right now the app is frontend-only: it imports fake data from a mockData.ts file, so "adding a resident" just changes a JavaScript array in memory and is lost on refresh. The whole productionization project is about inserting the middle box (backend) and the right box (database) so data becomes real and permanent.

3. Monorepo & the shared package

Monorepo = one git repository holding multiple projects (here: the frontend, the backend, and a shared library) that are developed together.
clinical-care/ ├─ apps/ │ ├─ web/ ← the React frontend (browser) │ └─ api/ ← the Express backend (server) └─ packages/ └─ shared/ ← types, validation schemas, permission rules used by BOTH web and api
Why a shared package? The frontend and backend must agree on what a "Resident" looks like. If they each defined it separately, they'd drift apart and cause bugs. By defining Resident once in packages/shared and importing it in both, there is a single source of truth. Same for validation rules and permissions.
npm workspaces = npm's built-in way to manage a monorepo. One npm install at the root installs everything and links the packages together (so import ... from "@clinical/shared" just works).

4. Frontend architecture

apps/web/src/ ├─ main.tsx ← entry point: mounts React into the HTML ├─ App.tsx ← defines all the routes (URL → page) ├─ pages/ ← one component per screen (Dashboard, Residents…) ├─ components/ │ ├─ ui/ ← generic building blocks (Button, Dialog, Table) │ ├─ layout/ ← Sidebar, Header, the page shell │ ├─ resident/ ← the 16 profile tabs │ └─ dashboard/ ← dashboard widgets ├─ context/ ← global state (who's logged in) ├─ hooks/ ← reusable bits of logic ├─ lib/ ← helpers (e.g. the future API client) └─ data/mockData.ts ← fake data (being replaced by the API)
Component = a reusable, self-contained piece of UI written as a function that returns JSX. <Button>, <ResidentProfile>, and a whole <Dashboard> page are all components. Small components compose into bigger ones.
JSX = the HTML-looking syntax inside React components (return <div>Hello</div>). It's JavaScript in disguise — you can put variables in with { }.

5. Components, props, state, hooks

Props — passing data in

Props = the inputs to a component, like function arguments. The parent passes them down.
// Parent passes residentId DOWN to the child
<ProgressNotesTab residentId="1" />

// Child receives it as props
function ProgressNotesTab({ residentId }: { residentId: string }) {
  // ...use residentId
}

State — data that changes over time

State = data a component "remembers" between renders. When state changes, React automatically re-draws the component. Created with the useState hook.
const [showForm, setShowForm] = useState(false);
//     ↑ current value  ↑ function to change it   ↑ initial value

<Button onClick={() => setShowForm(true)}>Add Note</Button>
{showForm && <NoteForm />}   // form shows only when showForm is true
Why not just a normal variable? A normal variable changing wouldn't tell React to redraw the screen. useState is the bridge: "when this value changes, update the UI."

Hooks — reusable logic

Hook = a special function (name starts with use) that lets a component tap into React features or share logic. Built-in ones: useState (state), useEffect (run code after render, e.g. fetch data), useContext (read global state). You can write your own — e.g. our future useResidents().
// useEffect: "after this component appears, do something"
useEffect(() => {
  document.title = "Residents";
}, []);  // [] = run once when the component first mounts

The render cycle

flowchart LR A[Component runs] --> B[Returns JSX] B --> C[React draws it] C --> D{State or props change?} D -->|yes| A D -->|no| E[Stays as-is]

6. Routing & navigation

Routing = deciding which component to show based on the URL. A Single-Page App (SPA) never reloads the page; the router swaps components in place, so navigation feels instant.
<Routes>
  <Route path="/" element={<AppLayout />}>      // shared shell (sidebar+header)
    <Route index element={<Dashboard />} />      // "/" shows Dashboard
    <Route path="residents" element={<Residents />} />
    <Route path="residents/:id" element={<ResidentProfile />} />  // :id = variable
  </Route>
</Routes>
  • :id is a URL parameter — /residents/3 makes id = "3", read with useParams().
  • Nested routes: everything inside AppLayout renders where its <Outlet /> is — that's how the sidebar/header stay put while the page content changes.
  • useNavigate() programmatically changes the URL (e.g. click a row → go to that resident).
The _redirects file (/* /index.html 200) exists because in an SPA the server only has one real HTML file. Without this rule, refreshing on /residents would 404. It tells the host "for any URL, serve index.html and let React Router sort it out."

7. Context & global state

Context = a way to share data with many components without passing props through every level ("prop drilling"). Perfect for things like "who is logged in" that lots of components need.
// 1. Create it
const UserContext = createContext(null);
// 2. Provide it high up (wraps the app)
<UserProvider>  <App />  </UserProvider>
// 3. Consume it anywhere below
const { name, role } = useUser();   // custom hook wrapping useContext
Today it's hardcoded (name: "Sarah Johnson"). In Phase 3 this becomes a real AuthContext that gets the logged-in user from the backend's /auth/me endpoint. Same pattern, real data.

8. Backend architecture (layers)

The API is organized in layers, each with one job. A request flows down through them and the response flows back up:

defines the URL + method

reads the request,
calls the service, sends response"] Co --> S["Service

touches the database (Prisma)"] Rep --> DB[(Postgres)]
LayerResponsibilityAnalogy
Route"This URL + HTTP method maps to this handler."The address on an envelope
MiddlewareCross-cutting checks that run before the handler (is this user allowed? is the input valid?).Security checkpoint
ControllerTranslates HTTP ↔ plain function calls. No business logic.Receptionist
ServiceThe actual rules ("a task must have an assignee").The specialist doing the work
RepositoryReads/writes the DB. The only layer that knows SQL/Prisma exists.The filing clerk
Why layers? Each can be changed or tested independently. Swap Postgres for another DB? Only the repository changes. Change a business rule? Only the service changes. This is "separation of concerns."
Middleware = a function that sits in the request pipeline and can inspect/modify the request, then either pass it along (next()) or stop it (send an error). Auth, RBAC, logging, and validation are all middleware.
// A middleware: block the request unless logged in
function requireAuth(req, res, next) {
  if (!req.user) return res.status(401).json({ error: "Not logged in" });
  next();  // ok, continue to the next layer
}

9. Database, Prisma & the ORM

Relational database = data in tables (rows & columns) with relationships between them. A resident has many progress notes → the notes table has a resident_id column pointing back ("foreign key").
ORM (Object-Relational Mapper) = a library that lets you work with database rows as normal objects in your language, instead of writing SQL strings. Prisma is our ORM.
// Without an ORM (raw SQL — error-prone, injection-risky):
db.query("SELECT * FROM residents WHERE facility_id = '" + id + "'");

// With Prisma (typed, safe, auto-parameterized):
prisma.resident.findMany({ where: { facilityId: id } });
Why an ORM matters for security. Prisma builds parameterized queries automatically, so a value like '; DROP TABLE residents; -- is treated as plain text, not SQL. That structurally prevents SQL injection — one of the oldest, most dangerous web vulnerabilities.

How Prisma works

  1. You describe your tables in schema.prisma (one place).
  2. prisma migrate turns that into real SQL tables + keeps a version history (migrations).
  3. prisma generate creates a fully-typed client, so prisma.resident. autocompletes every column.
// schema.prisma — describing a table
model Resident {
  id         String   @id @default(uuid())
  facilityId String
  name       String
  careLevel  CareLevel
  notes      ProgressNote[]   // one resident → many notes
  facility   Facility @relation(fields: [facilityId], references: [id])
}
Migration = a versioned change to the database structure. Like git commits, but for your table shapes. Lets the whole team (and production) evolve the DB safely and reproducibly.
Seeding = filling a fresh database with starter data. Our seed.ts loads the existing mockData.ts into Postgres — so the moment the DB exists, it already has the demo residents, and the frontend can switch over gradually.

10. A request's full journey

Follow "nurse adds a progress note" all the way through:

sequenceDiagram participant U as User (browser) participant Q as TanStack Query participant API as Express API participant MW as Middleware participant SVC as Service/Repo participant DB as Postgres U->>Q: clicks "Save Note" Q->>API: POST /residents/1/progress-notes {note, category} API->>MW: auth → is token valid? MW->>MW: RBAC → can this role create clinical? MW->>MW: zod → is the body valid? MW->>SVC: passes clean request SVC->>DB: prisma.progressNote.create(...) DB-->>SVC: new row SVC->>DB: audit_log.create(...) (who did what) SVC-->>API: created note API-->>Q: 201 Created + JSON Q->>Q: update cache, re-render list Q-->>U: new note appears instantly

Notice how many checks happen before any data is written: authenticated? authorized? valid? Only then does the write happen, and it's audited. That's defense in depth.

11. Authentication flow

Authentication = "who are you?" (login). Authorization = "what are you allowed to do?" (RBAC). Different questions — you're authenticated once, authorized on every action.
JWT (JSON Web Token) = a signed, tamper-proof string the server gives you at login that proves who you are. The server signs it with a secret key; if anyone edits it, the signature breaks and it's rejected. It's like a wristband at an event — hard to forge, checked at every door.
sequenceDiagram participant B as Browser participant API as API B->>API: POST /auth/login {email, password} API->>API: bcrypt.compare(password, storedHash) API-->>B: Set-Cookie: accessToken (httpOnly), refreshToken (httpOnly) Note over B,API: cookie is invisible to JavaScript (XSS-safe) B->>API: GET /residents (browser auto-sends cookie) API->>API: verify JWT signature → identify user → RBAC check API-->>B: data (or 401/403 if not allowed)
  • bcrypt.compare — checks a password against the stored hash without ever un-hashing (hashing is one-way).
  • httpOnly cookie — the browser stores & sends it automatically, but page JavaScript cannot read it, so an XSS attack can't steal the session.
  • access vs refresh token — the access token is short-lived (limits damage if leaked); the refresh token quietly gets a new access token so the user isn't logged out every 15 minutes.
  • 401 vs 403 — 401 = "I don't know who you are" (log in); 403 = "I know you, but you're not allowed."

12. Data fetching (TanStack Query)

TanStack Query = a library that handles the whole lifecycle of server data in React: fetching, caching, loading/error states, and re-fetching when data goes stale — so you don't hand-write all that plumbing.

The migration in one picture — this is the core of Phase 4:

// BEFORE (mock data, in-memory, lost on refresh):
import { progressNotes } from "@/data/mockData";
const notes = progressNotes.filter(n => n.residentId === residentId);

// AFTER (real data from the API, cached, always fresh):
const { data: notes = [], isLoading } = useProgressNotes(residentId);
//      ↑ same shape (thanks to shared types) — the JSX barely changes
// The hook (in services/), reused by any component:
function useProgressNotes(residentId) {
  return useQuery({
    queryKey: ["progress-notes", residentId],       // cache identity
    queryFn: () => api.get(`/residents/${residentId}/progress-notes`),
  });
}
// Writing data:
const createNote = useMutation({
  mutationFn: (body) => api.post(`/residents/${id}/progress-notes`, body),
  onSuccess: () => queryClient.invalidateQueries(["progress-notes", id]),
  //                 ↑ tells the cache "this data changed, re-fetch it"
});
queryKey = a unique label for a piece of cached data. Invalidation = marking cached data as stale so it re-fetches — that's how a newly-added note appears immediately without a manual refresh.
Strangler-fig migration. Because shared types keep the shape identical, we migrate one resource at a time. Un-migrated components keep reading mockData.ts; migrated ones hit the API. The app never fully breaks during the transition. (Named after a vine that gradually replaces the tree it grows on.)

13. Validation with zod

Schema validation = defining the exact shape data must have, and rejecting anything that doesn't fit — before it reaches your logic or database.
const createTaskSchema = z.object({
  title:      z.string().min(1, "Task description is required"),
  assignedTo: z.string().min(1),
  priority:   z.enum(["Low", "Medium", "High", "Urgent"]),
  dueDate:    z.string().optional(),
});
type CreateTaskInput = z.infer<typeof createTaskSchema>;  // free TypeScript type!
One schema, two jobs. The same zod schema lives in packages/shared and is used by (1) the frontend form (via react-hook-form) to show inline errors as the user types, and (2) the backend to reject bad/malicious requests. Rules can never drift out of sync, and z.infer gives you the TypeScript type for free.

15. Build it yourself: the checklist

If you wanted to recreate this from scratch, this is the order that keeps you sane:

  1. Scaffold the frontend — npm create vite@latest (React + TS). Add Tailwind + a couple of shadcn components. Build one static page.
  2. Add routing — install React Router, make 2–3 pages navigate.
  3. Build the UI with fake data — a mockData.ts file. Get the whole app looking right before any backend. (This is exactly where the project started.)
  4. Go monorepo — npm workspaces; extract shared types into packages/shared.
  5. Stand up Postgres — locally via Docker; define schema.prisma; migrate + seed from your mock data.
  6. Build the API — Express with the layer structure; one resource end-to-end first (residents), then copy the pattern.
  7. Add auth — login endpoint, bcrypt, JWT cookies, a ProtectedRoute on the frontend.
  8. Wire the frontend to the API — TanStack Query; migrate one resource at a time; delete mock arrays as you go.
  9. Harden — validation everywhere, RBAC, audit log, security headers, env validation.
  10. Ship — CI (lint+test+build), deploy frontend static + backend + managed Postgres.
Learning tip. Don't try to hold it all at once. The one idea that unlocks everything: the browser asks, the server decides, the database remembers. Every feature — notes, tasks, login, AI — is just another trip around that loop. Master one full loop (e.g. progress notes) and the other 20 are copies of it.

The app now has a real Postgres database mirroring all the demo data. Here's everything introduced, in learn-able detail.

New terms this phase

TermMeaning

How multi-tenancy is stored

Every tenant-owned table has a facilityId column. The tenant root is the Facility table; Staff and Resident point at it, and every clinical record points at a resident (and also carries facilityId for direct, index-friendly filtering).

Facility (1) ──< Staff (many)
Facility (1) ──< Resident (many) ──< ProgressNote / Task / Wound / … (many)
                                     each row ALSO stores facilityId
Why store facilityId on the leaf tables too (not just derive it through the resident)? Speed and safety. Every clinical query can filter WHERE facilityId = ? directly against a composite index, and the tenant guard works even on tables you reach without joining through Resident.

Proof it isolates tenants

After seeding one facility with 5 residents, a query scoped to a different facility id returns nothing:

SELECT count(*) FROM "Resident"
WHERE "facilityId" = '<some-other-facility>';
-- → 0   (this facility's data is invisible to others)

That single property, enforced on every query, is what lets you safely sell one deployment to many hospitals/homes.

Security introduced this phase

  • Passwords hashed with bcrypt (cost 12) even for seed users — never plaintext.
  • Env validated at boot (config/env.ts with zod): the server won't start without valid DATABASE_URL and JWT secrets. Real .env is gitignored; .env.example documents it.
  • Identifier columns (ihi, medicareCard) are isolated so they can be column-encrypted in the pre-pilot hardening step.
  • Audit log table created (append-only, no PII in body) ready for the API to write to.
Try it yourself. Run npm run prisma:studio -w @clinical/api and click through the tables — you'll see the exact residents from the mock UI, now living in Postgres. Change a name in Studio, refresh: it persisted. That's the difference between mock data and a real database.
