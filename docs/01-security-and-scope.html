System Overview & Threat Model

The platform stores Personal Health Information (PHI) of aged-care residents: names, dates of birth, diagnoses, medications, wounds, behaviour notes, and national identifiers (IHI, Medicare, DVA). Under Australian law this is "sensitive information" — the highest-protection category under the Privacy Act 1988 and the Australian Privacy Principles (APPs).

Who we protect against

Threat actorExample attackPrimary control
External attackerCredential stuffing, SQL injection, stolen JWTAuth + rate limiting + parameterized queries (ORM)
Malicious insiderStaff snooping on residents not in their careRBAC + audit log + least privilege
Curious insiderCarer viewing admin-only reportsRole-based UI + server-side authorization
Compromised deviceXSS stealing tokens from browser storagehttpOnly cookies (tokens unreadable by JS)
Data-at-rest breachStolen DB backup / diskEncryption at rest + column encryption for IDs
Golden rule. The frontend's role-based UI is a convenience, never a security control. Every permission decision is re-checked on the server. A user who forges a request must still be blocked by API-side RBAC.

1. Authentication DO NOW

Status: IMPLEMENTED (Phase 3). bcrypt password verification, JWT access+refresh tokens in httpOnly/Secure/SameSite=strict cookies, and /api/auth login·logout·refresh·me are live and verified (an unauthenticated request gets 401; a valid login sets cookies and returns the user). Deactivated accounts are locked out on the next request. Still to do (pre-pilot): account lockout, password-strength policy, and MFA for privileged roles.

Requirement (from scope): "Create system users, assign roles, activate/deactivate, password reset."

What we are doing

  • Email + password login. Passwords hashed with bcrypt (cost factor 12) — never stored or logged in plaintext.
  • On login, issue two JWTs: a short-lived access token (~15 min) and a long-lived refresh token (~7 days), both delivered as httpOnly; Secure; SameSite=Strict cookies.
  • POST /auth/login, /auth/logout, /auth/refresh, GET /auth/me.
  • Account activate/deactivate flag checked on every request (deactivated user = instant lockout).
Suggestion — go beyond the requirement. The scope only says "password reset," but for a clinical system I recommend: (1) account lockout after 5 failed attempts (throttle brute force); (2) password strength policy (min 12 chars, checked against the "Have I Been Pwned" breached-password API); (3) plan MFA (TOTP) for Admin/Care Manager roles before pilot — aged-care providers increasingly require it. These are cheap to add now and expensive to retrofit.
Why bcrypt and not plain hashing? MD5/SHA-256 are fast — an attacker can try billions of guesses/sec against a stolen DB. bcrypt is deliberately slow and salted, making each guess expensive. argon2id is an even stronger modern alternative worth considering.

2. Authorization (RBAC) & Multi-Tenancy DO NOW

Status: IMPLEMENTED (Phase 2). The shared permission map is enforced server-side by requirePermission middleware on every route, and verified: a Carer's attempt to create a resident returns 403 while an Admin's returns 201. Tenancy is enforced by taking facilityId from the session token (never the client) on every query — proven at the DB layer (cross-facility query → 0 rows) and now at the API layer. Resident-level care-team scoping remains a later enhancement.

Requirement: "Assign user roles" — four roles: Registered Nurse, Care Manager, Admin, Carer.

What we are doing

  • A shared permission map (packages/shared/src/permissions.ts) defines, per role, which actions (read/create/update/delete) are allowed on which resource (residents, clinical, tasks, documents, reports, staff, audit).
  • Express rbac middleware enforces it on every route before the controller runs.
  • The same map drives the frontend to hide buttons the user can't use.
Suggestion — add tenant isolation from day one. The current plan is single-facility. Because you told me you want to scale, I recommend building multi-tenancy now: every table carries a facilityId, and a middleware injects the logged-in user's facilityId into every query. This means one deployment can safely serve many aged-care homes with zero data bleed between them — and retrofitting tenancy later is one of the most painful migrations there is. I have baked this into the DB design in §11.
Suggestion — resident-level access scoping (later). RBAC answers "can nurses edit notes?" It does not answer "can THIS nurse edit THIS resident?" For a pilot with one ward that's fine. Before multi-ward scale, add a care_team assignment table so a carer only sees residents in their assigned wing — the "minimum necessary access" principle that auditors look for.

3. Data Protection DO NOW

Requirement: confidentiality banner — "handles personal and clinical data… do not share."

What we are doing

LayerControl
In transitHTTPS/TLS everywhere (HSTS header forces it)
At restManaged-Postgres disk encryption (AES-256) + encrypted upload volume
Data residencyDatabase + file storage pinned to AWS Sydney (ap-southeast-2)
In logsPII redaction — no names/IHI/Medicare ever written to application logs
Suggestion — column-level encryption for national identifiers. Disk encryption protects a stolen disk, but anyone with DB read access sees IHI/Medicare in plaintext. I recommend application-level encryption (or Postgres pgcrypto) specifically on ihi, medicareCard, and dvaCard columns, so even a DB dump leaks nothing usable. Decrypt only when displaying to an authorized user. This is a strong differentiator for aged-care buyers.
Data residency is a legal requirement, not a preference. Australian health data generally must stay onshore. This constrains hosting (§ hosting) and any AI model calls (§10) — you cannot send resident data to a US-only LLM endpoint.

4. Audit Logging & Accountability DO NOW

Status: IMPLEMENTED (Phase 2, mutations). An auditMutations middleware writes every successful create/update/delete to AuditLog with actor id, action, entity, a one-way-hashed IP, and facilityId — no PII in the body. Verified after live requests. Still to do: log read access (viewing a resident), make the table append-only at the DB level (revoke UPDATE/DELETE), and hash-chain rows for tamper-evidence.

Requirement: "User activity logs, record change history, access logs."

What we are doing

  • An append-only audit_log table records every mutating action: who (staff id), what (action + entity + entity id), when (timestamp), from where (hashed IP). No PII in the log body.
  • Written by Express middleware so no route can "forget" to log.
Suggestion — capture read access too, and make it tamper-evident. The requirement says "access logs," which for health data means logging views of a resident record, not just edits (auditors ask "who looked at Mrs X?"). Also: make the audit log append-only at the DB level (revoke UPDATE/DELETE grants) and consider hash-chaining each row to the previous one, so tampering is detectable. This is what turns "we have logs" into "we can prove what happened."

5. API & Transport Security DO NOW

Status: IMPLEMENTED (Phase 2). helmet, a strict cors allowlist (only the frontend origin, credentials enabled for cookies), a global rate limit, and a tighter /auth limiter are all wired in app.ts. A 1 MB JSON body cap guards against payload-bomb DoS. Still to do: custom CSP tightening and a WAF before pilot.
  • helmet — sets secure HTTP headers (CSP, HSTS, X-Frame-Options, etc.) in one line.
  • CORS allowlist — only the known frontend origin may call the API (no wildcard).
  • Rate limiting — express-rate-limit, tightest on /auth (defeats brute force).
Suggestion. Add a strict Content-Security-Policy (helmet default is permissive) to blunt XSS, and put the API behind a WAF (Cloudflare/AWS WAF) before pilot for automated bot/OWASP protection.

6. Input Validation & Injection Defense DO NOW

  • zod schemas (in packages/shared) validate every request body/params — shared with the frontend forms, so validation rules can't drift.
  • Prisma ORM parameterizes all queries → SQL injection is structurally prevented (no string-concatenated SQL).
  • React escapes output by default → stored-XSS is mitigated (never use dangerouslySetInnerHTML with resident data).
Suggestion. Add server-side output sanitization for any rich-text/free-text clinical notes if you ever render them as HTML, and set a max request body size to prevent payload-bomb DoS.

7. Secrets & Configuration DO NOW

  • All secrets (DB URL, JWT signing keys) in environment variables, never in git.
  • config/env.ts validates env with zod at boot — the server refuses to start if a secret is missing (fail-fast, no silent insecure defaults).
  • .env.example committed as documentation; real .env is gitignored.
Suggestion. Before pilot, move secrets to a managed secrets manager (AWS Secrets Manager / Doppler) with rotation, rather than plain env files on the server.

8. Session & Token Management DO NOW

Why httpOnly cookies, not localStorage? A token in localStorage is readable by any JavaScript — one XSS bug and the attacker steals the session. An httpOnly cookie is invisible to JS; the browser attaches it automatically and script can't read it. Combined with SameSite=Strict this also blocks CSRF. For PHI this is the correct choice.
Suggestion. Add refresh-token rotation (issue a new refresh token on each use, invalidate the old) and a server-side token denylist so logout/deactivation revokes access immediately, rather than waiting for the 15-min access token to expire.

10. AI Feature Security FUTURE

The source PDFs flag two AI features: Care Plan AI summary and Care Statements generation. Both send clinical data to a language model — the highest-risk data flow in the system.

Planned architecture

  • A separate AI service (never in the request path of clinical writes) that reads structured data + notes and drafts summaries.
  • Clinical text embedded with pgvector for semantic search / retrieval-augmented generation (RAG) over a resident's own history only.
  • Every AI output is human-reviewed before saving, and stored flagged as "AI-drafted / clinician-approved by X."
Non-negotiable AI controls. (1) Data residency — use an AU-region model endpoint (e.g. Azure OpenAI Australia East or AWS Bedrock Sydney); do not call a US-only API with resident data. (2) No training on your data — contractually ensure the provider won't retain/train on prompts. (3) Minimize the prompt — send only the clinical facts needed, strip direct identifiers where possible. (4) Log every AI call to the audit trail. (5) Never auto-save AI output into a clinical record without a clinician's explicit approval — the human is accountable, the AI is a drafting assistant.
Suggestion — scope AI narrowly at first. Start with the lowest-risk, highest-value use: summarizing a resident's recent progress notes into a handover paragraph. It's bounded, obviously useful to nurses, and easy to review. Defer anything that looks like clinical decision-making (risk scoring, medication suggestions) until you have clinical governance sign-off — that territory carries regulatory (TGA "software as a medical device") implications.

11. Database Design

Status: IMPLEMENTED (Phase 1). This design is now live in Postgres via Prisma (apps/api/prisma/schema.prisma). All tables below exist with facilityId + UUID keys + cascade deletes + composite indexes, seeded from the demo data (1 facility, 7 staff, 5 residents, all clinical children). Tenant isolation was verified: a query scoped to a different facility returns zero rows. The pgvector embedding column is deferred to the AI phase (local Postgres lacks the extension); everything else is built.

Postgres, normalized, multi-tenant. Facility is the tenant root; Resident is the clinical hub with many resident-scoped child tables. Staff and AuditLog are facility-scoped. Diagram below (simplified — child clinical tables all follow the same resident_id FK pattern).

Design decisions & recommendations

DecisionRationale / Recommendation
UUID primary keys (not auto-increment ints)Mock data uses "1", "2". Recommend UUIDs: non-guessable (can't enumerate residents by incrementing an ID), and safe to generate client-side / merge across facilities.
facility_id on every tableMulti-tenancy foundation. A single tenant middleware scopes all queries. Add a composite index (facility_id, resident_id) on every clinical table.
Allergies as their own tableMock nests them as a JSON array on Resident. Recommend normalizing to a table so they're queryable ("show all penicillin-allergic residents") and reportable.
embedding vector on notespgvector column reserved now (nullable) so the AI/RAG feature needs no schema migration later.
Time-series tables (vitals, BGL, notes)These grow fastest. Future: partition by month and archive old partitions to keep queries fast at scale.
Documents = metadata onlyStore the file in object storage (S3 Sydney); the DB row holds a storage_key, never the bytes. Cheaper, faster, and lets you encrypt/expire files independently.

Scaling plan FUTURE

  • Reads: add Postgres read-replicas + a Redis cache for hot lookups (resident lists, dashboards).
  • Writes: keep the API stateless behind a load balancer so you scale horizontally; the DB is the only stateful tier.
  • Big tables: time-based partitioning on vitals/notes; move cold data to cheaper storage.
  • Search: pgvector for semantic note search; add Postgres full-text or OpenSearch if keyword search across facilities is needed.
